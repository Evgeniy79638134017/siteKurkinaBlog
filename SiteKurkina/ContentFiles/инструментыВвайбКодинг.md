### Tailwind CSS v4 setup for Next.js

Tailwind v4 is a complete rewrite. **No `tailwind.config.js` needed.** CSS-first configuration only.

```bash
npm i tailwindcss @tailwindcss/postcss postcss
```

**postcss.config.mjs:**
```js
const config = {
  plugins: {
    "@tailwindcss/postcss": {},
  },
};
export default config;
```

**app/globals.css** ŌĆö the entire file:
```css
@import "tailwindcss";
```

That's it. No `@tailwind base/components/utilities` directives. No `postcss-import` or `autoprefixer` ŌĆö handled internally. Content detection is automatic. Custom themes go in CSS:

```css
@import "tailwindcss";

@theme {
  --color-primary: #6366f1;
  --color-secondary: #ec4899;
  --font-sans: "Inter", sans-serif;
}
```

### shadcn/ui setup

shadcn CLI v4 supports **both Radix UI and Base UI** as the underlying primitive layer.

```bash
# New project (scaffolds complete app)
npx shadcn@latest init

# Choose primitive library
npx shadcn@latest init --base radix   # Default ŌĆö Radix UI primitives
npx shadcn@latest init --base base    # Base UI primitives (MUI)

# Add components
npx shadcn@latest add button card dialog
npx shadcn@latest add -a              # Add ALL components
```

**React Aria** also has a hooks-only API via `react-aria` (`v3.47.0`) for granular control. Supports 30+ languages and 13 calendar systems.

## 9. Deployment (Vercel)

**Vercel CLI** latest version: **~39.x** (install: `npm i -g vercel`)

```bash
# Deploy
vercel                  # Preview deployment
vercel --prod           # Production deployment
vercel deploy --prebuilt # Deploy pre-built artifacts only

# Environment variables
vercel env add MY_KEY production    # Add env var
vercel env pull                     # Pull to local .env.local
vercel env remove MY_KEY            # Remove

# Link project
vercel link
```

**Auto-deploy from GitHub:** Import your repo at vercel.com ŌåÆ every push to a non-production branch triggers a **Preview Deployment** with unique URL. Merging to `main` triggers a **Production Deployment** automatically. PR comments show deployment links.

**Environment variables:** Dashboard ŌåÆ Project ŌåÆ Settings ŌåÆ Environment Variables. Select target environment (Production/Preview/Development). Client-side vars need `NEXT_PUBLIC_` prefix. Redeploy for changes to take effect.

## 16. Claude Code setup

### CLAUDE.md best practices

Place `CLAUDE.md` at your repo root (check into git). It's automatically pulled into context at the start of every Claude Code session.

**What to include:**
- Build/test/lint commands
- Code style rules (ESM, formatting, naming)
- Project structure and key files
- Stack and dependency info
- Workflow instructions (branch naming, testing requirements)
- Warnings and gotchas

**Placement hierarchy:**
1. `~/. claude/CLAUDE.md` ŌĆö global, all projects
2. `repo/CLAUDE.md` ŌĆö project root (most common)
3. `repo/CLAUDE.local.md` ŌĆö personal, gitignored
4. `repo/packages/foo/CLAUDE.md` ŌĆö scoped to subdirectory in monorepos

Use `/init` in Claude Code to auto-generate an initial CLAUDE.md. Press `#` during sessions to add instructions.

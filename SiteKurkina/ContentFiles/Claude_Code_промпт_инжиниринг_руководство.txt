# Мастерство Claude Code: полное руководство по промпт-инжинирингу

**Самая эффективная практика для Claude Code обманчиво проста: никогда не позволяйте ему писать код, пока вы не проверили план.** Этот воркфлоу «сначала план» в сочетании с грамотно составленным файлом CLAUDE.md и агрессивным управлением контекстом — то, что отличает разработчиков, получающих надёжный продакшн-результат, от тех, кто застревает в бесконечных циклах «починил — сломал». Это руководство объединяет официальную документацию Anthropic, личный воркфлоу создателя Claude Code и проверенные сообществом практики января–марта 2026 года — с конкретными настройками для стека Next.js + Tailwind CSS v4 + shadcn/ui.

---

## Воркфлоу «исследуй, планируй, кодируй, коммить» меняет всё

Борис Черни (создатель Claude Code в Anthropic) и все опытные практики сходятся в одном: **Claude сразу бросается писать код без исследования и планирования, и результат страдает катастрофически.** Официальный воркфлоу Anthropic состоит из четырёх фаз:

### Фаза 1 — Исследование

Попросите Claude прочитать релевантную кодовую базу, не написав ни строчки. Будьте явными: *«Прочитай файлы в src/auth/ и разберись, как мы обрабатываем сессии. НЕ пиши код.»* Используйте настойчивые формулировки: «глубоко», «детально», «со всеми особенностями» — без этого Claude читает поверхностно. Попросите Claude записать выводы в файл `research.md`, чтобы вы могли проверить его понимание перед продолжением.

### Фаза 2 — Планирование

Попросите Claude создать детальный план реализации. Используйте **триггеры мышления** для выделения большего бюджета рассуждений: `"think"` < `"think hard"` < `"think harder"` < `"ultrathink"`. Каждое ключевое слово даёт Claude прогрессивно больше времени на размышления.

«Цикл аннотаций» Бориса Тане — техника с наивысшим рычагом по мнению сообщества: после того как Claude запишет план в markdown-файл, откройте его в редакторе, добавьте инлайн-правки и ограничения прямо в документ, затем скажите Claude: *«Я добавил заметки в план — учти все заметки и обнови. Пока не реализуй.»* Повторите 1–6 раз. Постоянный артефакт (файл) переживает компактификацию; инструкции только из чата — нет.

### Фаза 3 — Код

Только после одобрения плана: *«Реализуй всё. Когда завершишь задачу, отметь её выполненной в документе плана. Не останавливайся, пока все задачи не будут выполнены.»* Для сложных фич реализуйте инкрементально: один файл → запустить тесты → следующий файл.

### Фаза 4 — Коммит

Используйте `/commit` для автогенерации conventional commit-сообщений или цепочку операций через кастомные слэш-команды. **Коммитьте после каждого значимого изменения** — AI может вносить регрессии, а гранулярные коммиты делают откат тривиальным.

Встроенный **Plan Mode** в Anthropic (активируется через `Shift+Tab` дважды) принудительно запрещает запись файлов на уровне инструментов, предотвращая написание кода во время планирования. Борис Черни использует Plan Mode и итерирует с Claude до удовлетворения, затем переключается в режим автопринятия, где Claude обычно реализует всё «с одного захода».

## CLAUDE.md — ваш файл с максимальным рычагом

CLAUDE.md — единственный файл, гарантированно находящийся в контекстном окне каждой сессии, и единственный источник инструкций, который **переживает компактификацию**. Когда Claude достигает лимита контекста и автоматически компактифицирует, инструкции из разговора исчезают — но CLAUDE.md перечитывается с диска. Это делает его фундаментом всего.

### Оптимальный размер: 60–80 строк

Официальное руководство Anthropic говорит — до 200 строк, но тестирование сообщества показывает, что следование инструкциям значительно падает после 80 строк. Системный промпт Claude Code оборачивает CLAUDE.md пометкой, что содержимое «может быть, а может не быть релевантным» — то есть Claude активно игнорирует контент, который считает неважным. Каждая строка должна оправдывать своё присутствие. Задайте себе вопрос: *«Если убрать это, Claude будет делать ошибки?»* Если нет — удаляйте.

### Рекомендуемая структура (в порядке важности):

- **Обзор проекта** (1–2 строки): что это за проект
- **Тех-стек с конкретными версиями**: фреймворк, язык, стилизация, библиотека компонентов
- **Команды**: сборка, тесты, линтинг, dev-сервер (самый универсально полезный раздел)
- **Критические правила**: инструкции «НИКОГДА» и «ВСЕГДА» с акцентом через `ВАЖНО` или `ОБЯЗАТЕЛЬНО` — но экономно, потому что если всё помечено как важное, ничто не важно
- **Архитектура**: ключевые директории и их назначение
- **Конвенции кода**: только то, что Claude не может вывести из существующего кода
- **Подводные камни и предупреждения**: специфичные для проекта ловушки и обходные пути

### Что НЕ включать:

- Секреты и API-ключи
- Правила стиля кода, которые обрабатывают линтеры («никогда не отправляйте LLM делать работу линтера»)
- Очевидные знания о фреймворке, которые Claude и так знает
- Инструкции по личности («будь senior-инженером»)
- Обширную документацию (лучше дайте ссылку на доки)
- Сниппеты кода — они быстро устаревают; используйте ссылки на файлы

### Иерархия файлов

Файлы загружаются в определённом порядке:

1. `~/.claude/CLAUDE.md` — глобально для всех проектов (до **15 строк**)
2. `CLAUDE.md` в корне проекта — основной файл, коммитится в git
3. `CLAUDE.local.md` — личные переопределения, добавляется в .gitignore
4. `CLAUDE.md` в дочерних директориях — загружаются по требованию, когда Claude работает в этих папках

### Паттерн прогрессивного раскрытия

Держите CLAUDE.md компактным, но давайте Claude доступ к глубокому контексту:

```markdown
## Справочная документация
- При изменении схемы БД: прочитай `docs/database_schema.md`
- Для паттернов Tailwind v4: прочитай `docs/tailwind-v4-guide.md`
- Для API-паттернов: прочитай `docs/api-architecture.md`
```

Команда `/init` генерирует стартовый CLAUDE.md, анализируя кодовую базу. Используйте как отправную точку, затем агрессивно редактируйте — удалите общие рекомендации, добавьте инструкции по воркфлоу, которые Claude не мог бы вывести сам, и сократите до 80 строк.

## Структура промптов, стабильно дающая хороший код

Каждый эффективный промпт для Claude Code следует формуле: **роль + задача + контекст + ограничения + формат вывода + критерии успеха.** Но конкретные паттерны сильно зависят от типа задачи.

### Для создания компонентов — ссылайтесь на существующий код:

```
Создай компонент NotificationSettings, следуя тому же паттерну, 
что и app/settings/profile/page.tsx. Серверный компонент с экспортом 
metadata, используй компонент PageHeader из components/ui, создай 
клиентский компонент для интерактивных элементов. Используй 
существующий enum NotificationType из types/notifications.
```

Один конкретный пример из вашей кодовой базы стоит 100 слов объяснений. Бенчмарки Anthropic показывают **~65% улучшение консистентности**, когда у Claude есть референсный файл для следования.

### Для отладки:

Предоставьте полный стек-трейс, 20–50 строк окружающего кода и детали окружения. Попросите и быстрый фикс, и долгосрочное решение в формате git diff. Для визуальных багов вставьте скриншот — он передаёт проблему быстрее любого описания.

### Для рефакторинга — всегда требуйте план миграции перед изменениями кода:

```
Прочитай текущую реализацию аутентификации в src/auth/ сначала. 
Создай план: какие файлы меняются, в каком порядке, потенциальные 
breaking changes, необходимые зависимости. НЕ пиши код, пока я 
не одобрю план.
```

### Для TDD — будьте предельно явными:

```
Строгий TDD:
1. Напиши падающие тесты для [фича] на основе ожидаемых input/output
2. Запусти тесты и убедись, что они ПАДАЮТ
3. Закоммить тесты
4. Жди моего одобрения
5. Напиши код, чтобы тесты прошли — НЕ модифицируй тесты
Без моков. Только реальные интеграции.
```

### Защитный промпт (guardian prompt)

Предотвращает изменение кода за пределами текущей задачи. Включите в CLAUDE.md или в начало промпта:

```
ОБЛАСТЬ: Модифицируй только файлы в src/api/auth/
НЕ ТРОГАЙ: tests/, config/, node_modules/
Ссылайся, но не модифицируй: authentication.md
Делай только те изменения, которые были запрошены или явно необходимы.
Исправление бага НЕ требует рефакторинга окружающего кода.
НЕ добавляй фичи, не рефактори код и не делай «улучшения» 
сверх того, что было запрошено.
```

Собственный системный промпт Anthropic включает антиинжиниринговые указания: *«Не создавай хелперы, утилиты или абстракции для одноразовых операций. Не проектируй под гипотетические будущие требования.»* Модели Claude 4.x воспринимают инструкции буквально — если хотите дополнительного усилия, явно запросите его.

## Управление контекстным окном предотвращает 90% проблем с качеством

**Деградация контекста — главный режим отказа Claude Code.** Контекстное окно на 200K токенов заполняется содержимым файлов, выводом инструментов и историей разговора. Качество падает задолго до достижения лимита из-за проблемы «потерянного в середине» — информация, похороненная в длинных разговорах, депри-оритизируется.

### Ключевые правила:

**Используйте `/clear` между каждой несвязанной задачей.** Это самая высокоэффективная привычка по контексту. Старая история тратит токены и может сбивать Claude. Чистая сессия с лучшим промптом почти всегда лучше длинной сессии с накопленными коррекциями.

**Используйте `/compact` вручную при 70% использования контекста**, не дожидаясь автокомпактификации при 95%. Укажите кастомные инструкции: `/compact сосредоточься на изменениях API и списке модифицированных файлов`. Отслеживайте использование через `/context`, чтобы видеть точное распределение токенов.

**Используйте `/rewind` (или `Esc Esc`) вместо споров с Claude.** После двух неудачных коррекций одной проблемы прекратите попытки исправления в контексте. Откатитесь к чекпоинту или полностью `/clear` и напишите лучший начальный промпт, учитывая полученный опыт.

**Субагенты — ваше оружие для сохранения контекста.** Каждый субагент работает в собственном изолированном контекстном окне и возвращает в основную сессию только краткое резюме. Делегируйте исследования, чтение файлов и верификацию субагентам, чтобы основной контекст оставался чистым. Определяйте их в `.claude/agents/` как markdown-файлы с YAML frontmatter.

**Паттерн HANDOFF.md** сохраняет прогресс между сбросами контекста: перед очисткой длинной сессии попросите Claude написать файл передачи, документирующий цель, что было попробовано, что сработало, что не сработало и следующие шаги. Следующая сессия читает этот файл и продолжает бесшовно.

## Файлы правил, навыки, хуки и слэш-команды — многослойная система

Помимо CLAUDE.md, Claude Code предлагает иерархию инструментов настройки, каждый со своей функцией:

### Файлы правил (`.claude/rules/`)

Разделяют инструкции на фокусированные файлы с привязкой к путям. Используйте YAML frontmatter для указания целевых директорий: файл `frontend/components.md` с `paths: ["src/components/**"]` загружается только когда Claude работает в этой директории. Это держит контекст компактным, предоставляя глубокие указания где нужно.

### Навыки (`.claude/skills/`)

Предоставляют знания по запросу, которые не нужно загружать в каждую сессию. Claude видит описания навыков при старте сессии, но загружает полное содержимое только при необходимости. Идеально для специфичных для библиотек знаний — паттерны Tailwind v4 или конвенции shadcn/ui, которые важны только при работе над UI.

### Хуки (hooks)

Выполняются детерминированно в точках жизненного цикла — в отличие от инструкций CLAUDE.md, которые являются рекомендательными. Хук `PostToolUse`, запускающий `prettier --write` после каждого редактирования файла, гарантирует форматирование. Хук `PreToolUse` на `Bash(git commit*)`, запускающий тест-сьют, блокирует коммиты при падении тестов. Хуки — слой принудительного исполнения, которым CLAUDE.md быть не может.

### Кастомные слэш-команды (`.claude/commands/`)

Хранят переиспользуемые шаблоны воркфлоу как markdown-файлы. Борис Черни использует слэш-команды для каждого повторяющегося воркфлоу: коммиты, PR, упрощение кода, верификация. Команды поддерживают `$ARGUMENTS` для параметризации и инлайн bash (`!git status`) для динамического контекста. Коммитьте их в git, чтобы вся команда получала пользу.

### Ментальная модель слоёв:

- **CLAUDE.md** = что применяется в каждой сессии
- **Правила** = конкретные политики с привязкой к путям
- **Навыки** = специализированные знания, активируемые по намерению
- **Хуки** = детерминированное принуждение
- **Слэш-команды** = переиспользуемые воркфлоу

## Антипаттерны, порождающие сломанный код

### Эффект «whack-a-mole» (починил — сломал)

Исправление одной вещи ломает другую. Причина — исчерпание контекстного окна. Когда 80% контекста занято чтением файлов и выводом инструментов, Claude теряет отслеживание ограничений. Решение: маленькие диффы (один файл → тест → следующий файл), субагенты для изолированной работы, `/clear` между задачами и хуки, блокирующие коммит при падении тестов.

### Цикл коррекций

Второй по частоте ловушка. Claude делает что-то неправильно, вы поправляете, всё ещё неправильно, поправляете снова — каждая коррекция добавляет шум в контекстное окно. **После двух неудачных коррекций всегда `/clear`** и напишите лучший начальный промпт. Никогда не спорьте со сбитым с толку Claude.

### Размытые мега-промпты

Надёжно производят плохой код. *«Добавь тесты для foo.py»* даёт генерические тесты. *«Напиши тест-кейс для foo.py, покрывающий крайний случай, когда пользователь разлогинен, используя реальные фикстуры базы данных, не моки»* — даёт именно то, что нужно. Конкретность — техника промптинга с наивысшим рычагом.

### Раздутый CLAUDE.md

Парадоксально снижает качество. Когда CLAUDE.md превышает ~200 строк, Claude начинает выборочно игнорировать инструкции. Сообщество сообщает, что **60% обращений в поддержку Claude Code** приходят из проектов вообще без CLAUDE.md — но вторая по частоте проблема — файлы CLAUDE.md настолько длинные, что критические правила теряются.

### Тяжёлое использование MCP

Может убить производительность. Если определения MCP-инструментов потребляют более **20K токенов**, они съедают контекст, который должен обслуживать вашу реальную работу. Отключайте неиспользуемые MCP-серверы через `/mcp`.

### Отсутствие цикла верификации

Оставляет Claude производить правдоподобно выглядящий, но некорректный код. Всегда предоставляйте критерии успеха — тесты, скриншоты, ожидаемый вывод — чтобы Claude мог проверить собственную работу. Anthropic утверждает, что это «единственное действие с наивысшим рычагом»: *«Выход Claude значительно улучшается при итерации. Первая версия может быть хорошей, но после 2–3 итераций обычно выглядит намного лучше.»*

## Продвинутые техники для опытных пользователей

### Agent Teams

Экспериментальная функция (включается через `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1`), позволяющая запускать параллельных агентов, координируемых Team Lead. Каждый участник команды имеет собственное контекстное окно и может работать независимо. Лучше всего подходит для: параллельного исследования конкурирующих подходов, кросс-слойной работы (фронтенд/бэкенд/тесты у разных участников) и отладки с конкурирующими гипотезами. Стоимость — примерно **3–4× токенов** от одной сессии.

### Git worktrees

`claude --worktree feature-name` создаёт изолированные рабочие директории для параллельных сессий. Каждое worktree имеет собственную ветку, директорию и git-индекс. Комбинируйте с Agent Teams для максимальной параллельности. Борис Черни запускает **5 параллельных локальных экземпляров Claude**, каждый в своём git checkout, плюс 5–10 удалённых сессий — и ожидает, что **10–20% сессий будут заброшены** из-за неожиданных результатов. Это нормально.

### Паттерн fan-out

Использует headless-режим для пакетных операций: Claude генерирует список задач, затем вы в цикле вызываете `claude -p "мигрируй foo.py с React на Vue. Верни OK или FAIL." --allowedTools Edit Bash(git commit:*)`. Отточите промпт за несколько запусков.

### Хуки как движки воркфлоу

Подход Ника Тьюна по hook-driven разработке (февраль 2026) использует хуки Claude Code как события, управляющие детерминированным воркфлоу с сохранением состояния. Хуки могут отклонять вызовы инструментов, нарушающие доменные инварианты — решая проблему принуждения, с которой рекомендательные правила CLAUDE.md не справляются.

## Tailwind v4 + shadcn/ui: преодоление критического разрыва в знаниях

Это **самая значительная проблема** при использовании Claude Code с данным стеком. Обучающие данные Claude содержат сотни тысяч примеров Tailwind v3, тогда как кодовые базы v4 только появляются. Без явных указаний Claude генерирует паттерны v3 примерно **в 80% случаев** — создаёт файлы `tailwind.config.js`, использует `@tailwind base/components/utilities` и импортирует устаревшие плагины.

### Критические различия, которые Claude путает:

| Паттерн v3 (по умолчанию Claude) | Правильный паттерн v4 |
|---|---|
| `tailwind.config.js` / `.ts` | CSS-first конфиг в `globals.css` через `@theme { }` |
| `@tailwind base; @tailwind components; @tailwind utilities;` | `@import "tailwindcss";` |
| `require('@tailwindcss/typography')` в JS-конфиге | `@plugin "@tailwindcss/typography"` в CSS |
| `theme.extend.colors` в JS | `@theme { --color-primary: #6f53b8; }` в CSS |
| `tailwindcss-animate` | `tw-animate-css` (для совместимости с shadcn v4) |
| `bg-opacity-50` | `bg-black/50` (CSS color modifiers) |
| `@layer base { :root { } }` | `:root { }` на корневом уровне (не внутри `@layer base`) |

### Архитектура CSS для shadcn/ui + Tailwind v4

Четырёхшаговая обязательная структура:

1. Определите CSS-переменные с обёрткой `hsl()` в `:root`
2. Свяжите переменные с Tailwind-утилитами в `@theme inline { }`
3. Примените базовые стили в `@layer base`
4. Используйте семантические цветовые классы типа `bg-background`, которые автоматически переключаются в тёмной теме без `dark:` вариантов

В файле `components.json` поле `tailwind.config` должно быть **пустой строкой** — это говорит CLI shadcn использовать CSS-first подход v4. Стиль `default` устарел — используйте `new-york`. Компонент `toast` устарел — используйте `sonner`.

### Практическое решение

**Установите навык tailwind-v4-shadcn** из реестра Tessl:

```bash
npx tessl i github:secondsky/claude-skills --skill tailwind-v4-shadcn
```

Тестирование сообщества показывает переход от **~65K токенов и 2–3 ошибок к ~20K токенов и 0 ошибок** — улучшение эффективности на 70%.

**Next.js 16.2+** теперь автогенерирует `AGENTS.md` и включает связанную документацию в `node_modules/next/dist/docs/`. Добавьте `@AGENTS.md` в ваш CLAUDE.md, чтобы перенаправить Claude с устаревших обучающих данных на документацию, соответствующую версии.

## Готовый CLAUDE.md для Next.js + Tailwind v4 + shadcn/ui

```markdown
# Проект: [Название]

## Тех-стек
- Next.js 15+ (App Router, React 19, Server Components)
- TypeScript (strict mode)
- Tailwind CSS v4 (CSS-first конфиг — НИКАКОГО tailwind.config.js)
- shadcn/ui (стиль new-york)
- pnpm

## Команды
- `pnpm dev` / `pnpm build` / `pnpm lint` / `pnpm test`
- `npx shadcn@latest add [компонент]` — добавить shadcn-компонент

## КРИТИЧНО: Tailwind CSS v4
- НИКАКОГО tailwind.config.js — вся настройка в globals.css через @theme
- Используй `@import "tailwindcss"`, НЕ директивы @tailwind
- Используй директиву `@plugin` для плагинов
- Используй `tw-animate-css`, НЕ tailwindcss-animate
- Используй модификаторы цвета (bg-black/50), НЕ opacity-утилиты
- Container queries встроены (плагин не нужен)

## Правила shadcn/ui
- Импорт из @/components/ui/[компонент]
- Используй cn() из @/lib/utils для условных классов
- Используй варианты компонентов, не кастомные переопределения цветов
- Используй sonner для тостов (компонент toast устарел)

## Конвенции Next.js
- По умолчанию Server Components; "use client" только при необходимости
- Server Actions для мутаций, Route Handlers для API
- Всегда используй next/image и next/link
- Params в page-компонентах — промисы (await)

## Стиль кода
- Named exports, без default exports
- Никаких типов `any`; деструктуризация пропсов
- Mobile-first responsive: sm:, md:, lg:, xl:
- Группировка импортов: react → next → библиотеки → локальные (@/*)

## Воркфлоу
- Сначала ПЛАН, потом КОД
- Маленькие диффы: один файл → тесты → следующий файл
- Запускай typecheck после изменений кода

## Справочные документы
- Паттерны Tailwind v4: прочитай docs/tailwind-v4-guide.md
- Паттерны компонентов: следуй конвенциям src/components/ui/
```

## Заключение

Разрыв между посредственным и отличным выводом Claude Code сводится к трём вещам: **дисциплина планирования**, **гигиена контекста** и **явные знания о стеке**.

Цикл «исследование → план → аннотации → реализация» — не опциональная нагрузка, а механизм, предотвращающий дорогие циклы переработки, которые тратят больше времени, чем само планирование. CLAUDE.md — не файл документации, а артефакт промпт-инжиниринга, который нужно итерировать как любую критическую часть инфраструктуры. Для Tailwind v4 конкретно — разрыв в знаниях реален и достаточно велик: без явных правил v4 в вашем CLAUDE.md вы потратите больше времени на коррекцию Claude, чем на ручное кодирование.

Разработчики, которые выпускают быстрее всего с Claude Code, не пишут хитрые промпты — они строят **системы** из файлов CLAUDE.md, навыков, хуков и слэш-команд, которые делают правильный вывод путём наименьшего сопротивления.
